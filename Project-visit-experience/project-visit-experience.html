The Problem
Skilled nursing adds complexity the current visit structure couldn't handle.
Current system had no clinical data model, so vitals tracking and assessments had nowhere to live.
The product was built for sparse visits, but skilled nursing needed dense, continuous documentation.
I took a design decision:
I stopped organizing the visit around system categories and aligned it to how caregivers actually work:
context → observation → assessment → documentation → submission.

The Solution
How this plays out across the visit
01. Facesheet
The facesheet is the center of the visit. Client info, care goals, allergies, and active medications surfaced upfront. This is where real-time clinical decisions happen.
Caregivers should never have to search for context they need in the first 30 seconds of a visit.
Facesheet UI (annotated)
High-frequency information surfaced upfront client info, goals checklist, allergies, medications
02. Medications at the point of care
Medications are managed in real time during the visit. Caregivers review and log actions directly in the facesheet, without leaving their workflow. This pattern scaled to support interaction checks and reminders.
Progressive disclosure for dense data
Medication lists are dense and time-sensitive. The accordion keeps the list compact by default, expands on demand, and supports inline actions in both states.

What this structure enabled
In-visit logging
Medication actions are logged without leaving the facesheet. The visit flow stays intact.
Progressive disclosure
The accordion surfaces what caregivers need when they need it, without exposing all detail at once.
Structural flexibility
When interaction checks and reminders were added later, the layout didn’t need to change. The structure absorbed the new complexity.
03. Vitals
Vitals posed a different problem. Not where to surface them, but how to capture them without sacrificing data quality. Free text entry produced inconsistent data with no way to compare or surface patterns across visits. Phase 2 replaced it with structured inputs: each vital has its own field with contextual sub-inputs (position, site, systolic vs. diastolic) and a notes field for edge cases. The constraint was speed. Structured inputs had to be at least as fast as typing.

The remaining tabs follow the same principle: one task, one place.
The structure reduces overlap and keeps each task focused. No tab tries to do too much each one has a clear scope within the visit, making the overall system easier to navigate and maintain.
Body System Exam
Questionnaires scoped to specific body systems: respiratory, cardiovascular, neurological. Caregivers assess what is clinically relevant, not everything at once.
Incident Log
Timestamped, structured entries for clinical events: falls, acute illness, hospitalization. Unstructured incident notes are unusable downstream.
Wound Care
Wound status captured with structured inputs for size, stage, and treatment applied. Consistent fields make documentation traceable across visits.
Notes
Free-form documentation for observations that don’t fit structured fields. Every structured system needs an escape valve.
Forms & Documents
Clinical forms and reference documents kept within the visit context. Leaving the app to access a form breaks the visit flow.
GPS
Location verification for visit start and end. Carried forward from the original system because the underlying need didn’t change even though the visit did.

WHAT CHANGED
What the structure made possible
Most of the gains came from structural decisions, not from any individual screen. When the visit is organized around how caregivers work, the friction disappears before the UI is even considered.
Context switching eliminated mid-visit
Caregivers previously navigated across tabs and external tools to piece together a visit. Grouping clinical tasks by workflow stage meant the visit could be completed in one continuous sequence without breaking flow to hunt for context.
Medication logging moved inline
Surfacing medications on the facesheet with inline actions meant caregivers could log “given / not given” without leaving the active visit. Documentation happened where the clinical action happened.
High-density data made workable
Free text vitals and flat medication lists created noise, not records. Structured inputs scoped to each vital type, combined with progressive disclosure on medications, made dense clinical data capturable without slowing caregivers down.
The structure absorbed post-launch additions
Medication interaction checks and reminders were integrated after launch without restructuring the visit. When new requirements arrived, the system had a place for them. That was the real test of whether the architecture was sound.
REFLECTION
What this project clarified
Skilled nursing forced a shift from screen-level thinking to system-level thinking. The lessons weren’t subtle.
IA decisions outweigh UI decisions
The most impactful work had nothing to do with visual design. Organizing the visit around caregiver tasks rather than system categories shaped everything else downstream.
Flexibility is a design decision, not an afterthought
Requirements shifted throughout execution. Designing each component to hold a range of scope, not a fixed one, meant the system stayed usable as things changed.
Clinical environments are not controlled flows
Caregivers work fast, under pressure, mid-visit. Every decision was tested against those conditions, not an idealized use case.
A good system absorbs new requirements
Interaction checks, reminders, and new assessment types were added after launch. None of them required rethinking the visit structure. That was the real test.
