STARTING FROM ZERO
GEOH did not support medications. To enable skilled care, the system needed a way to store, review, manage, and use medication data during a live visit.
Where data is created
The client profile — medication data is entered, structured, and managed here. This is the source of truth for what a patient is taking.
Where data is used
The visit interface — medication data is surfaced and acted on in real time during a caregiver visit. The same data must be reliable in both places.
This meant designing both where data is created and where it is used — and making sure those two surfaces stay in sync.
FOUNDATION — STRUCTURING MEDICATION DATA
Medication data is dense. Each entry includes dosage, frequency, route, status, dates, and notes. The challenge was: make this data glanceable without losing detail.
Grouping related inputs into readable units
Instead of exposing every field separately, I grouped related inputs. For example: dosage = quantity + unit + frequency — shown as one entity. Medications are added through a structured modal, but displayed in a scan-friendly grid.

Once medications were structured and visible, the next problem surfaced.
How do we ensure these medications are safe together? Drug interactions are complex: they change when medications change, they involve multiple entities, and they require decisions — not just visibility.
The goal shifted from showing interaction data to ensuring users always see accurate, current interaction information — in context. Instead of separate reports or external checks, I designed interactions as part of the medication system itself.

MAKING INTERACTIONS RELIABLE
Eight design decisions that ensure the interaction system stays accurate, consistent, and useful over time — not just at the moment data is entered.
1 Interactions as states, not static results
Interactions are dynamic — they change as medications are added, updated, or removed. Rather than treating them as a one-time check, I modelled them as a state that evolves with user decisions.
Interaction found → Pending review → Approved
State diagram
Interaction state evolves with user decisions
2 Keep decisions valid over time
If medications change, existing approvals may no longer be valid. The system resets affected interactions to Pending, forcing re-review. This prevents outdated approvals from persisting silently.
State reset after medication change
Prevents outdated approvals
3 One interaction = one source of truth
Interactions affect multiple medications. The system ensures no duplication — one interaction record is shared across all linked medication entries. One decision updates all related records consistently.
Linked interaction updates
One decision updates all related medications
4 Visibility at the right level
Each medication in the grid shows: total interactions, and the count of pending vs approved. Users can understand status at a glance without opening details — quick understanding without deep navigation.
Interaction counts in grid
Quick understanding without deep navigation
5 Detail without overload
Interaction details are expandable via accordion inside the modal. This progressive disclosure pattern allows users who need full detail to access it without forcing everyone to see it by default.
Interaction modal accordion
Handles dense content efficiently
A Critical Constraint
6 Handling unreliable identifiers
The interaction system depended on medication data from an external API. A key limitation: the API did not provide a reliable unique identifier for each medication.
The problem

Without stable identifiers, interaction results could not be safely mapped over time — updates could lead to inconsistencies and previous checks could become unreliable.

The decision

Instead of persisting interaction mappings, I designed the system to recompute interactions whenever the medication list changes.

System behaviour
  • Adding a new medication triggers a full interaction check
  • Interaction states are recalculated for the entire list
  • Affected approvals are reset to Pending
Why this works

Ensures interaction data always reflects current medications. Avoids incorrect mappings. Keeps the system consistent despite API constraints.

Interaction re-check trigger
System revalidates interactions on every change
7 System-driven triggers
Two mechanisms keep interaction data current: a manual "Run Interaction Check" action available to the user, and an automatic check triggered on every medication change. Together they ensure interaction data is always up to date — regardless of whether the user remembers to run a check.
8 Edge case handling
To maintain consistency at the edges, I documented each state transition scenario — state resets, partial approvals, linked updates — and created shared edge case documentation with the engineering team. Each scenario includes the trigger, the expected system state, and the corresponding UI state.
EXTENDING INTERACTIONS INTO THE VISIT
Interactions are not only reviewed in the profile grid. They are surfaced in the visit accordion — where actions happen.
Why this matters
The same truth available in the profile is available where decisions are executed. Caregivers don’t need to leave the visit to check interaction status.
One system, two surfaces
Profile grid and visit accordion share the same underlying interaction state — so a decision made in the profile is reflected immediately during the next visit.

EXTENSION — SCHEDULING AND REMINDERS
Once interaction logic was stable, the system extended into execution. Medications became time-based actions during a visit.
Time-based prompts
Instead of showing full schedules, prompts appear ~30 minutes before scheduled time and are visible only during visit hours — reducing noise while surfacing what matters.
Notifications
Triggered only within the visit window and grouped when multiple medications are due close together — minimising interruption while maintaining clinical accuracy.

REFLECTION
The core challenge was not UI
This project evolved from storing medication data to maintaining a system where data, logic, and actions stay aligned over time. The real design work was defining how interaction data behaves, how decisions stay valid, and how the system remains consistent across surfaces — not designing individual screens.
Constraints shaped better decisions
The API's unreliable identifiers forced a recompute-on-change approach that, in retrospect, made the system more robust than a mapping-based approach would have been. Working within a hard constraint produced a simpler and more reliable solution.
